# ajolote

Shared AI agent configuration for multi-tool teams.

`ajolote` keeps a single canonical config (`.agents/config.json`) and translates it into the native format of each developer's AI coding assistant ‚Äî Claude Code, Cursor, Windsurf, GitHub Copilot, Cline, and Aider. Tool-specific files are gitignored; only the shared config is committed.

## Install

```sh
curl -fsSL https://raw.githubusercontent.com/antoumpas/ajolote-ai/main/install.sh | sh
```

## Usage

```sh
# Maintainer ‚Äî one-time setup in any project (Python, Go, Java, Ruby ‚Äî anything)
ajolote init

# Developer ‚Äî after cloning, generate config for your tool of choice
ajolote use claude   # or cursor, windsurf, copilot, cline, aider

# Two-way sync ‚Äî pull in changes from your tool, push updated config back out
ajolote sync           # syncs all tools whose files are present
ajolote sync cursor    # syncs only cursor

# Check what's generated locally
ajolote status
```

## How sync works

`ajolote sync` runs in two directions:

**‚Üë Import** ‚Äî reads MCP server definitions from the tool's own config files and merges any new ones into `.agents/config.json`. This catches servers a developer added directly in their tool's UI.

**‚Üì Export** ‚Äî regenerates the tool's files from the (now updated) canonical config, so rules and context are always current.

| Tool | Importable (‚Üë) | Exported (‚Üì) |
|---|---|---|
| claude | `.claude/settings.json` | `CLAUDE.md`, `.claude/settings.json` |
| cursor | `.cursor/mcp.json` | `.cursor/rules/agents.mdc`, `.cursor/mcp.json` |
| cline | `.roo/mcp.json` | `.clinerules`, `.roo/mcp.json` |
| windsurf | ‚Äî | `.windsurf/rules/agents.md` |
| copilot | ‚Äî | `.github/copilot-instructions.md` |
| aider | ‚Äî | `.aider.conf.yml` |

Rules always flow **config ‚Üí tool** only. `.agents/config.json` is the authority for rules; the tool files are never trusted for rules.

## config.json reference

`.agents/config.json` is the only file you edit. Everything else is generated from it.

```jsonc
{
  // Project metadata ‚Äî used as context in every generated file
  "project": {
    "name": "my-app",
    "language": "TypeScript",
    "stack": "TypeScript / Node.js / React",
    "repoType": "monorepo",
    "testRunner": "Vitest"
  },

  // MCP servers shared across all tools that support MCP
  "mcp": {
    "servers": {
      "filesystem": {
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-filesystem", "."],
        "description": "Read/write access to this repo"
      },
      "github": {
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-github"],
        "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" },
        "description": "GitHub API ‚Äî issues, PRs, branches"
      }
    }
  },

  // Agent rules ‚Äî applied to all tools
  "rules": {
    "general": [
      "Read before writing. Always read the relevant files before modifying them.",
      "Minimal diffs. Change only what is necessary to satisfy the task."
    ],
    "codeStyle": [
      "Follow the existing conventions in each file. When in doubt, run the linter.",
      "Match the surrounding indentation, quote style, and semicolon usage exactly."
    ],
    "testing": [
      "Every new function or module must have at least one accompanying test.",
      "Do not delete or weaken existing tests."
    ],
    "security": [
      "Never hard-code secrets, tokens, or credentials ‚Äî use environment variables.",
      "Never log sensitive user data."
    ],
    "commits": [
      "Write commit messages in the imperative mood: Add X, Fix Y, Remove Z.",
      "One logical change per commit."
    ]
  },

  // Skill files ‚Äî reusable task instructions in .agents/skills/
  "skills": [
    ".agents/skills/git.md",
    ".agents/skills/testing.md"
  ],

  // Personas ‚Äî role-based behaviours agents should adopt for specific tasks
  "personas": [
    ".agents/personas/reviewer.md",
    ".agents/personas/architect.md"
  ],

  // Context ‚Äî background knowledge about the project (keep these up to date)
  "context": [
    ".agents/context/architecture.md",
    ".agents/context/data-model.md",
    ".agents/context/glossary.md"
  ]
}
```

### Fields

| Field | Description |
|---|---|
| `project.name` | Project name ‚Äî appears as a heading in generated files |
| `project.language` | Primary language (e.g. `TypeScript`, `Python`, `Go`) |
| `project.stack` | Full stack description (e.g. `Python / FastAPI / PostgreSQL`) |
| `project.repoType` | `monorepo` or `polyrepo` |
| `project.testRunner` | Test command or runner name (e.g. `pytest`, `Vitest`, `go test`) |
| `mcp.servers` | MCP server definitions ‚Äî translated to each tool's MCP config format |
| `mcp.servers.<name>.command` | Binary to run (e.g. `npx`, `node`) |
| `mcp.servers.<name>.args` | Arguments passed to the command |
| `mcp.servers.<name>.env` | Environment variables (use `${VAR}` to reference shell env) |
| `mcp.servers.<name>.description` | Human-readable description (optional) |
| `rules.general` | General agent behaviour rules |
| `rules.codeStyle` | Code style and formatting rules |
| `rules.testing` | Testing conventions |
| `rules.security` | Security constraints |
| `rules.commits` | Commit message conventions |
| `skills` | Paths to skill files ‚Äî reusable task instructions |
| `personas` | Paths to persona files ‚Äî role-based agent behaviours |
| `context` | Paths to context files ‚Äî background knowledge about the project |

## The .agents/ directory

All files in `.agents/` are committed to git and shared across the team. Agents receive these as context when developers run `ajolote use <tool>`.

```
.agents/
  config.json              ‚Üê canonical config (the only file ajolote reads)
  skills/
    git.md                 ‚Üê Git workflows: branch naming, PR hygiene
    testing.md             ‚Üê How to write and run tests
    api-design.md          ‚Üê REST / GraphQL conventions (add your own)
    migrations.md          ‚Üê DB migration workflow (add your own)
    debugging.md           ‚Üê Step-by-step debugging protocol (add your own)
  personas/
    reviewer.md            ‚Üê How to behave during code review
    architect.md           ‚Üê How to approach design and architecture tasks
  context/
    architecture.md        ‚Üê High-level system design (keep up to date!)
    data-model.md          ‚Üê Core entities and relationships
    glossary.md            ‚Üê Domain terms and abbreviations
```

`ajolote init` seeds all three directories with starter templates. Fill them in with your project's real information and commit ‚Äî that's where the value is.

### Skills

Reusable task instructions. Reference them under `"skills"` in `config.json`:

```json
"skills": [
  ".agents/skills/git.md",
  ".agents/skills/testing.md",
  ".agents/skills/api-design.md"
]
```

### Personas

Role-based behaviours agents adopt for specific task types. Reference under `"personas"`:

```json
"personas": [
  ".agents/personas/reviewer.md",
  ".agents/personas/architect.md"
]
```

Example usage ‚Äî a developer tells their AI tool:
> "Act as the architect persona and review this proposed change to the auth service."

The agent reads `.agents/personas/architect.md` and applies its mindset and principles.

### Context

Background knowledge about the project ‚Äî architecture, data model, glossary. Reference under `"context"`:

```json
"context": [
  ".agents/context/architecture.md",
  ".agents/context/data-model.md",
  ".agents/context/glossary.md"
]
```

Keep these files accurate. They are the most important inputs for agents working on non-trivial tasks.
